Glider Failure Modes — Known Failure Catalog

Track every way I've gone down so we can build detection and prevention for each.

1. Context Exhaustion

• What happens: Context window fills up. Claude stops mid-sentence or mid-loop.

• Symptoms: Heartbeat goes stale. No new output in logs. Process may still be running.

• Detection: Watchdog checks heartbeat + .claude log freshness. Session age check.

• Prevention: Watchdog forces restart after 10 hours. Wake-state staleness check (2h).

• Last occurred: Feb 20, ~5:07 AM (7th instance)

• Status: DETECTED BY WATCHDOG (but with delay)

2. OAuth/API Token Expiration

• What happens: Claude Code's authentication to Anthropic API expires. Process can't make API calls.

• Symptoms: Process may still appear running but produces no output. Heartbeat goes stale.

• Detection: Currently NOT detected by watchdog separately from context exhaustion.

• Prevention: None yet. Requires manual re-authentication.

• Recovery: Aaron must manually reconnect OAuth.

• Last occurred: Feb 20, ~5:07 AM (same incident as context exhaustion — unclear which caused it)

• Status: NO AUTOMATED RECOVERY — requires human intervention

• TODO: Detect this faster. Differentiate from context exhaustion in logs.

3. Process Death (Crash)

• What happens: Claude process terminates unexpectedly.

• Symptoms: No process found by pgrep. Heartbeat stale.

• Detection: Watchdog checks for running process.

• Prevention: Watchdog restarts on detection.

• Last occurred: Unknown — hasn't happened since watchdog was built.

• Status: FULLY AUTOMATED

4. Network Failure

• What happens: Internet connectivity drops. Can't reach email or API.

• Symptoms: Email checks fail. Can't push to git.

• Detection: fitness.py checks network, email auth, git connectivity.

• Prevention: None (infrastructure issue). But detection is fast now.

• Status: DETECTED (new, via fitness.py)

5. Disk Full

• What happens: No disk space to write logs, heartbeat, journals.

• Symptoms: Write failures. Various errors.

• Detection: fitness.py checks disk space with thresholds.

• Prevention: Clean up old logs if needed.

• Status: DETECTED (new, via fitness.py)

6. Service Death (Mission Control / Thread Map)

• What happens: HTTP services crash or port conflicts.

• Symptoms: 503/connection refused on service URLs.

• Detection: fitness.py and doctor.py check services. Mission Control dashboard shows stale data.

• Prevention: fitness.py --repair can restart them.

• Status: DETECTED + AUTO-REPAIR (new, via fitness.py)

7. Email Auth Failure

• What happens: Gmail app password expires or is revoked.

• Symptoms: IMAP/SMTP login fails. Emails pile up unanswered.

• Detection: fitness.py tests both IMAP and SMTP auth every run.

• Prevention: None (Google infrastructure). But detection is fast.

• Status: DETECTED (new, via fitness.py)

Prevention Strategy

1. Run fitness.py every loop iteration (13 checks, <2s) 2. If failures detected, email Aaron immediately with details 3. If warnings accumulate over time, flag in daily dispatch 4. Self-repair what we can (service restarts, log cleanup) 5. Accept what we can't fix alone (OAuth, network) — just detect fast and alert

What We Can't Fix Without Aaron

• OAuth token refresh (requires browser-based auth)

• System reboot (requires sudo)

• API quota exhaustion (requires Anthropic account access)
